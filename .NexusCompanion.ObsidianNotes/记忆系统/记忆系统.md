这个记忆系统有不少值得讲的创新点。我给你整理成一个面试时容易讲清楚的结构：

---

## 核心叙事：「它不是RAG，是认知架构」

面试官通常对 Agent 记忆的理解停留在「对话历史+向量检索」，你可以先一句话定调，然后展开亮点：

---

### 1. 双模存储：既给机器看，也给人看

**大多数系统二选一**：要么纯向量库（机器友好、人看不懂）、要么纯文件（反之）。Nexus **同时维护**两者：
- **Markdown 层**（MEMORY.md、HISTORY.md、journal/*.md）← 人可以直接翻看、修改，作为 source of truth
- **SQLite + sqlite-vec 层** ← 实时检索用

**讲给面试官**：这解决了两个问题——(1) Debug 时可以直接翻 Markdown 文件看记忆状态，不需要查数据库；(2) Markdown 层可以作为备份，向量库重建时可以重新 embed。相当于**用写时复制的一致性代价，换取了运维可见性**。

---

### 2. Hotness 感知检索（不是纯语义相似度）

传统 RAG 只看 $cos(q, d)$。这个系统引入了 **hotness 分 = 强化频次 × 时间衰减 × 情感权重**，最终分数是：
$$score = (1-\alpha) \cdot semantic + \alpha \cdot hotness$$

**关键设计**：情感权重高的记忆，半衰期自动延长（最长 +50%），意味着用户情绪强烈的事件会更持久地保持"热度"。

**讲给面试官**：从认知科学角度看，人的记忆不是平等排列的——高频回忆的和情绪强烈的事情更容易被想起。这个设计是在模拟这种效应，比纯语义相似度更贴近实际使用场景。

---

### 3. LLM 门控检索（QueryRewriter）

每次查记忆前，先用一个轻量 LLM 做两件事：
- 判断这次需不需要查记忆（RETRIEVE / NO_RETRIEVE）
- 如果需要，把原始 query 重写为更适合匹配 procedure/preference 的形式

**讲给面试官**：不是每轮对话都需要历史记忆，比如用户说"你好"就不需要检索。这个门控避免了大量无效的 embed + 检索开销。而且 query 重写把"具体问题"转成"通用模式"，提高了 procedure 匹配的召回率。

---

### 4. HyDE（假设文档嵌入）增强检索

先让 LLM 根据用户问题**生成一条假设性的记忆条目**，然后用这条假设记忆去向量库检索，再把原始 query 和 HyDE 的结果做 **max-pooling 合并**。

**讲给面试官**：用户不会用"记忆库里的语言"提问。HyDE 的作用是把"用户的问题"转译成"记忆应该有的样子"，显著提高了检索质量。而且这个实现做了 graceful degradation——HyDE 生成失败时只走原始路径，不会崩。

---

### 5. 后响应异步失效检测

用户说了"不对"、"别这样做了"之后，`PostResponseMemoryWorker` 会：
1. 从用户消息中提取"invalidation topics"
2. 扫描已有的 procedure/preference 条目
3. 匹配上的自动标记为 superseded

**讲给面试官**：这是在线学习中最难的问题之一——如何优雅地覆盖错误记忆。大多数系统要么直接覆盖（丢失历史），要么永远不覆盖（错误持续）。我们的方案是**做 supersede 而非 delete**，保留审计链路的同时，让新行为生效。

---

### 6. 情感权重作为一等公民

`emotional_weight` (0-10) 影响三件事：
- Hotness 衰减半衰期
- Profile 的替换阈值（高情感的记忆更难被覆盖）
- 记忆提取时 LLM 会主动标注

**讲给面试官**：这是把认知科学里的"情绪唤醒度"概念工程化了。用户的"不要这样"和"帮我查个天气"权重显然不同，系统应该区分对待。

---

## 面试话术模板

> 我觉得我们这个记忆系统最大的特点，是不把记忆当做一个"搜索问题"来解，而是当做一个**认知架构问题**。纯 RAG 的思路是 user query → 向量检索 → 拼接 context → 喂给 LLM，但我们做了几层不同的设计：
>
> 第一，记忆不是一块铁板，我们分了 **procedure / preference / event / profile** 四种类型，每种类型的存储策略、替换策略、阈值都不一样。
>
> 第二，检索不是单纯的余弦相似度，我们加了 hotness 分，让高频使用和情感强烈的记忆更容易被召回。
>
> 第三，写入链路上有 LLM 参与——不仅提取事实，还会生成 trigger tag、rule schema、情感权重，让每条记忆都是"结构化"的。
>
> 第四，我们有异步的**后响应处理管道**，能检测用户的纠正行为并自动 supersede 旧记忆，这是很多系统没有做到的在线学习能力。
>
> 最后，整套引擎是 plugin 架构，底层存储可以热替换，这也为后续扩展不同记忆后端留了接口。
