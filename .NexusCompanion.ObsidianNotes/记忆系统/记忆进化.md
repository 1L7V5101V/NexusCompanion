
## 四大记忆进化机制（从代码到面试表达）

### 1. 强化 (Reinforcement)

**核心逻辑**：`memory2/store.py:401-421`

```
content_hash + memory_type 双重去重
  → 命中则 reinforcement += 1
  → 若之前已 superseded，复活 + 强化
```

有两种触发路径：
- **显式写入**（`upsert_item`）← 同一条内容重复写入时，自动 `reinforcement+1`
- **语义去重**（`memorizer.py:239-247`）← consolidation 写入 event 时，cosine ≥ 0.92 且 7 天内，不写新条目而是 `reinforce_items_batch`

**hotness 公式**（`store.py:161-182`）：
```
freq = sigmoid(log1p(reinforcement))
recency = exp(-ln2 / effective_half_life × age_days)
hotness = freq × recency
effective_half_life = 14d × (1 + 0.5 × emotional_weight / 10)
# ↑ 情感权重高 → 半衰期最长延长 50%
```

最终检索分 = $(1-\alpha) \cdot \text{semantic} + \alpha \cdot \text{hotness}$

**面试讲法**：
> 强化机制解决的是"低频但重要"的记忆被埋没的问题。传统 RAG 只看语义相似度，用户反复确认过的事情和一次性事件分值一样。我们的做法是 content_hash 和语义去重两条路径都能触发 reinforcement +1，然后再通过 sigmoid(log1p) 变换到 hotness 分，与语义分做线性融合。这样用户叮嘱过三次的事，即使语义匹配度略低，也比一次性的高语义匹配更容易被召回。

### 2. 退休 (Retirement)

**核心逻辑**：`memorizer.py:112-156`

写新 memory 时，按 type 执行不同规则：

| Type | 阈值 | 特殊逻辑 |
|------|------|----------|
| procedure | ≥ 0.90 supersede | ≥ 0.70 且 same tool_requirement → 合并 |
| preference | ≥ 0.90 supersede | — |
| profile | ≥ 0.90 supersede | category=purchase/status 时；高情感(≥7) 提高到 0.92 |
| event | — | 不做 supersede（语义去重走 reinforce） |

**实现**（`store.py:552-565`）：
```python
# 退休只是状态标记，不是物理删除
UPDATE memory_items SET status='superseded', updated_at=? WHERE id=?
```

**替换审计**（`store.py:71-88`）：`memory_replacements` 表完整记录 `old_item_id → new_item_id`、双方全文、`relation_type`，支持撤销和审计。

**面试讲法**：
> 退休不是删除。我们只把旧条目标为 `superseded`，保留了完整的历史。优点有三个：第一，离线审计时可以追溯"这条规则之前是什么"；第二，undo 插件可以一键恢复——把 superseded 改回 active 就行；第三，不同类型的退休阈值不一样，profile 里情感权重高的事信息需要更高相似度才能被覆盖，避免情绪强烈的记忆被意外冲刷掉。

### 3. 合并 (Consolidation)

合并有两层意义，面试时一定要分开讲：

#### 3a. Procedure 合并 — `memorizer.py:270-286`

```python
_pick_explicit_merge_target(similar, extra, merge_threshold=0.70):
  # 只有相同 tool_requirement 的 procedure 才会合并
  if item.tool_requirement == new_item.tool_requirement:
     return item  # 选为目标
```

确认合并目标后调用 `merge_item()`（`store.py:1467-1531`），原子操作：
- `summary` 用 `_merge_summary_text()` 拼接（`旧；新`，去重冗余）
- `content_hash` + `embedding` 同步更新
- `reinforcement +1`
- 若 content_hash 冲突则安全降级为 supersede + upsert

#### 3b. Event 语义去重 — `memorizer.py:224-247`

consolidation 写入 event 时先查 7 天内的相似向量：
```
cosine ≥ 0.92 → 不写入，reinforce 已有条目
cosine < 0.92 → 作为新 event 写入
```

#### 3c. 文件层 Consolidation — `core/memory/markdown.py`

两个阶段的原子操作：
```
Phase 1:  extract from conversation → PENDING.md (暂存缓冲)
Phase 2:  when enough accumulated → LLM 提取事实
           → 写 HISTORY.md（不可变日志）
           → 写 RECENT_CONTEXT.md（压缩版）
           → 触发 ConsolidationCommitted 事件
```

**面试讲法**：
> 这里的 Consolidation 有两层。文件层是把零散对话累积到一定量后，一次性做 LLM 提取，抽成 profile/preference/procedure 写入记忆库；向量层则是带语义去重的 event 写入。Procedure 的合并有个很有意思的点——我们不用纯向量相似度判断，而是看 `tool_requirement` 是否相同，只有绑定同一个工具的操作流程才合并，这样避免把语义相似但工具不同的逻辑搞混。

### 4. 溯源 (Provenance)

溯源有两层实现：

#### 4a. 注入记忆的 `[item_id]` 标记 — `retriever.py:280-389`

每条注入的 memory 都带 `[item_id]` 前缀：
```
- [mem_abc123] 用户偏好用简洁的列表呈现（有印象，不确定）
- [mem_def456] 必须调用工具：web_search（必须调用工具：web_search）
```

这是给 LLM 看的"内部引用"，LLM 需要在回复末尾输出 `§cited:[...]§`。

#### 4b. Citation 插件 — `plugins/citation/plugin.py`

这是完整的引用协议，三个模块流水线：
```
1. PromptRender 时写入协议规则（LLM 知道要输出 §cited:[]§）
2. AfterReasoning 时解析 §cited:[mem_1,mem_2]§ → 提取到 cited_memory_ids
3. 从回复中剥离协议标记（对用户不可见）
```

如果 recall_memory 工具调用链中有 `cited_item_ids`，也会被 fallback 解析。

**`cited_memory_ids` 的应用**（`proactive_v2/test_post_guard_ack.py:49-52`）：
```python
delivery_key = sha1(json.dumps(sorted(cited_ids)).encode()).hexdigest()[:16]
# 引用相同的记忆 → 去重，避免重复推送
```

**面试讲法**：
> 溯源的核心是 `[item_id] → §cited:[]§` 的引用链路。注入记忆时每条都带 id 前缀，LLM 引用时在回复末尾输出协议行，citation 插件在后处理阶段解析、剥离、存入消息的 `cited_memory_ids` 字段。这个 ID 还会被 proactive 推送系统用来做 delivery key——如果两条推送引用了相同记忆，就知道是重复内容，可以直接跳过。溯源不是监控人的手段，是让系统自己知道"我为什么这么说"，从而支持去重、审计和记忆纠错。

### 四个机制的关系图

```
写入路径：
  ↑ 退休 (cosine≥0.90) → 旧条目: status=superseded + 记录到 memory_replacements
  │
新条目 ──→ 合并 (same tool_req≥0.70) → merge target: summary 拼接 + reinforcement+1
  │
  ↓ 强化 (content_hash 命中) → reinforcement+1 + 复活(若已退休)

检索路径：
  向量检索 → semantic_score
                          → score = (1-α)·semantic + α·hotness
  hotness ← 强化次数 × 时间衰减 × 情感权重

输出路径：
  [item_id] 注入 → LLM → §cited:[id1,id2]§ → 解析 → cited_memory_ids
                                                         ↓
                                              proactive 去重 + 审计
```
