简历里的 **prepare → render → reason → commit** 是对 7 阶段生命周期的**四步简化**，分别对应：

---

### 1. Prepare（会话准备）← Phase 1 BeforeTurn + Phase 2 BeforeReasoning

```python
# Phase 1: 获取 session、构建 ContextBundle
before_turn = await self._before_turn.run(state)       # 行 356

# Phase 2: 记忆检索、工具发现、prompt warmup
before_reasoning = await self._before_reasoning.run(...)  # 行 370
```

具体做的事：
- `SessionManager` 获取/创建 session，加载历史消息
- `ContextStore.prepare()` 调 **retrieval pipeline**（向量搜索 + HyDE + RRF 融合召回），产出 `ContextBundle`
- **工具发现**：ToolDiscoveryState 的 LRU 缓存决定本轮暴露哪些工具（`always_on` + LRU 命中的 `deferred`）
- 所有插件 Phase 注入点（`@on_before_turn`、`@on_before_reasoning`）在此触发

---

### 2. Render（构建提示词）← PromptRender

```python
prompt_render = await self.render_prompt(                # 行 800
    PromptRenderInput(session_key, history, memory_block, ...)
)
```

调用 `ContextBuilder.render()`，把各部分拼成最终的 `messages` 列表发给 LLM：
- System Prompt（身份、规则、工具 schema）
- Memory Context（MEMORY.md + PENDING.md 注入）
- Recent Context（近期对话摘要）
- Turn Injection（本轮消息、skill mention、hint）
- History（窗口化历史消息）

插件通过 `@on_prompt_render` 可以往任意 section 插入内容。

---

### 3. Reason（推理 + 工具执行循环）← Reasoner.run() 内部循环

```python
result = await self.run(initial_messages, ...)           # 行 821

# 内部 while True 循环（行 961）：
#   BeforeStep (token 估算 + 插件校验)
#   → LLM.chat() (带工具 schema)
#   → 如果是 tool_call: 执行工具 → AfterStep → 继续循环
#   → 如果是文本回复: break
```

这是最核心的部分，内部又是一个**三步骤循环**：

| 循环内步骤 | 代码 | 说明 |
|---|---|---|
| **BeforeStep** | `行 969` | token 估算、校验是否超限、插件可 `early_stop` |
| **LLM 推理** | `行 1012` | 调 DeepSeek API，传入 tools schema，获取 text/tool_calls |
| **工具执行 + AfterStep** | `行 1028-1114` | 逐个执行 tool_call，`ToolExecutor.preflight()` 做 hook 检查，`AfterStep` fanout |

关键设计点：
- **多轮 retry 机制**：如果 LLM 返回 ContentSafetyError 或 ContextLengthError，自动缩小 history window、裁剪 sections 重试（`_build_attempt_plans`，行 778）
- **Tool Search 元工具**：如果当前 visible 工具不足，LLM 可以调 `tool_search` 按需发现新工具
- **渐进式解锁**：`deferred` 工具首次调用时回填错误，LLM 学会先调 `tool_search` 再使用

---

### 4. Commit（持久化 + 派发）← Phase 5 AfterReasoning + Phase 6 AfterTurn

```python
# Phase 5: parse LLM 输出、持久化 session、构建出站消息
after_reasoning = await self._after_reasoning.run(...)   # 行 407

# Phase 6: TurnCommitted 事件 → 内存固化 + dispatch
return await self._after_turn.run(...)                   # 行 412
```

具体流程：
- **Parse**：提取 LLM 回复中的 thinking、tool traces
- **Persist**：保存 session（history trim、last_consolidated 标记）
- **Outbound**：构建 `OutboundMessage`（channel/chat_id/content/thinking/media）
- **TurnCommitted fanout**：触发内存 consolidation（LLM 从对话提取结构化事实→写入 PENDING.md + HISTORY.md）
- **Dispatch**：通过 `OutboundPort` 发送回 Telegram/QQ/CLI

---

### 面试深挖点

**Q：为什么设计 4 步而不是 7 步暴露在简历上？**
A：7 阶段是对内的 SPI（Service Provider Interface），4 步是对外的 API。简历上写 4 步是为了让面试官先建立整体认知，追问时再展开到 PhaseModule 拓扑排序的深度。

**Q：reason 步骤的 retry 机制具体怎么工作的？**
```python
# 行 778-885: 三个 attempt plan
attempts = [
    {"name": "full",         "history_window": full,  "disabled_sections": []},
    {"name": "half",         "history_window": half,  "disabled_sections": ["memory"]},
    {"name": "minimal",      "history_window": 0,     "disabled_sections": ["memory", "skills"]},
]
```
逐级降级：先全量→裁剪记忆块→只剩本轮消息。每一级都重新 render prompt 再调 LLM，ContextLengthError 时自动切下一级。

**Q：render 时 memory block 怎么拼进 prompt？**
A：`PromptRenderInput` 带 `retrieved_memory_block`（来自 Phase2 的向量检索结果），`ContextBuilder` 按 `[system_preamble, memory_context, recent_context, history, turn_injection]` 顺序拼接。每个 section 都是可选的，由 trim plan 控制是否启用。

**Q：commit 阶段的 memory consolidation 和 optimizer 有什么区别？**
A：consolidation（每次对话后）→ 追加写 PENDING.md；optimizer（每 18h）→ 全量重写 MEMORY.md。前者频次高只改小文件，后者频次低才改大文件，中间层 PENDING.md 保护 prompt cache 命中率。
