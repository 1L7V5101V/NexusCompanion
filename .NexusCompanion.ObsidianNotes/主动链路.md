主动链路就是：**用户没有主动发消息时，Agent 自己定期醒来，检查外部信息源和用户状态，判断要不要主动推送一条消息**。

它对应项目里的：

- `ProactiveLoop`
- `AgentTick`
- `Sensor`
- `DataGateway`
- `AnyActionGate`
- `TurnOrchestrator`
- `Drift`

可以把它理解成一个“后台值班系统”。

**1. 主动链路要解决什么问题**
传统 AI 助手是：

```text
用户问 -> AI 答
```

主动链路想做的是：

```text
系统定期醒来
 -> 看用户现在适不适合被打扰
 -> 看有没有重要信息
 -> 判断这条信息值不值得主动说
 -> 如果值得，就推送
 -> 如果不值得，就跳过或执行后台任务
```

比如：

- GitHub issue 有新的高优先级评论
- 飞书日历快到会了
- RSS/网页内容里出现用户关心的信息
- 用户已经很久没互动，可以低频提醒
- 用户刚刚聊天结束，就不要马上打扰

所以它不是“定时发消息”，而是“定时判断是否应该发消息”。

**2. 主动链路整体流程**
可以画成这样：

```text
ProactiveLoop 定时 tick
        |
        v
Sensor 感知用户状态
        |
        v
DataGateway 并行拉取 alert/content/context
        |
        v
多层门控判断
        |
        v
LLM 判断是否值得主动触达
        |
        v
去重 / 配额 / 防打扰
        |
        v
TurnOrchestrator 统一发送
        |
        v
写入 proactive session / trace / delivery state
```

更口语化一点：

> 主动链路每隔一段时间醒来一次，先判断用户现在能不能被打扰，再拉取外部数据源，然后让模型判断有没有值得说的内容。如果没有，就不发；如果有，就通过统一的消息出口推送给用户，并记录这次主动行为，避免重复打扰。

**3. ProactiveLoop：主动链路的心跳**
`ProactiveLoop` 是主动链路的主循环。

它做两件关键事：

1. 控制下一次什么时候醒来。
2. 每次醒来执行一次 `_tick()`。

不是固定 1 分钟或 5 分钟跑一次，而是根据分数动态调整。

简历里写的“自适应轮询”就是这里。

```text
活跃度高 / 信息多 / 最近上下文丰富
 -> tick 间隔变短

刚刚聊完 / 没有新内容 / 不适合打扰
 -> tick 间隔变长
```

面试可以说：

> ProactiveLoop 不是传统 cron，而是一个状态感知的后台循环。它根据用户活跃度和信息流状态动态计算下一次 tick 时间，从而平衡信息新鲜度和打扰成本。

**4. Sensor：感知用户状态**
`Sensor` 负责回答这些问题：

- 用户上次什么时候说话？
- 最近对话多不多？
- 之前主动推送用户有没有回复？
- 最近 24 小时已经推送了多少次？
- 用户是否可能在睡觉？
- 当前 session 是哪个用户/渠道？

这些信息会影响“可打扰度”。

比如：

```text
用户刚刚回复过主动消息
 -> 说明用户可能愿意继续互动

用户很久没回复主动消息
 -> 降低打扰欲望

24小时内已经主动推送很多次
 -> 疲劳度上升，减少推送

用户刚发完消息
 -> 不要马上主动插话
```

这就是主动链路比普通定时任务高级的地方：它不是只看“有没有内容”，还看“现在适不适合说”。

**5. 电量模型：量化社交节奏**
你简历里这段是主动链路的核心亮点：

```text
E(t)=a·exp(-t/T1)+β·exp(-t/T2)+γ·exp(-t/T3)
```

它的含义是：

> 用户刚互动完时，社交“电量”高；时间越久，电量逐渐衰减。

三个时间尺度：

```text
30min  -> 对话余温
4h     -> 当天语境
48h    -> 关系连续性
```

比如：

- 刚聊完 5 分钟：用户可能还在当前上下文里，但也不适合频繁插话。
- 过了 4 小时：当天语境还在，可以适度主动提醒。
- 过了 48 小时：关系连续性变弱，需要更谨慎地触达。

代码里还有一个反向指标：

```text
D_energy = 1 - energy
```

可以理解为“互动饥饿度”：

```text
energy 越低
 -> 越久没互动
 -> D_energy 越高
 -> 系统越倾向于检查是否需要主动触达
```

**6. 三维打分：不只看用户状态**
主动链路不只看用户多久没说话，还看内容和上下文：

```text
base_score = w_e · D_energy + w_c · D_content + w_r · D_recent
```

三个部分：

```text
D_energy  -> 用户多久没互动
D_content -> 外部信息源有没有新内容
D_recent  -> 最近对话上下文是否丰富
```

举例：

- 用户很久没说话，但外部没有新内容：不一定发。
- 外部来了很多新内容，但用户刚刚被打扰过：不一定发。
- 用户最近聊过某个主题，外部刚好来了相关内容：更值得发。

所以主动链路的判断是多因素综合，而不是单一规则。

**7. DataGateway：三路并行数据预取**
主动链路会从 MCP 数据源拉信息，但信息被分成三类：

```text
alert   -> 高优先级告警
content -> 普通内容流
context -> 背景上下文
```

这就是简历里的“三路并行数据预取”。

三类含义：

**alert**
高优先级，偏“必须看”。

比如：

- GitHub issue 被点名
- CI 挂了
- 日历会议即将开始
- 服务告警

这种信息通常不需要复杂筛选，优先级最高。

**content**
普通内容流，偏“可能有价值”。

比如：

- RSS 新文章
- GitHub trending
- 飞书文档更新
- 某个网页源的新内容

这类内容会先取 meta，比如标题、来源、URL，必要时再 fetch 正文。

**context**
背景上下文，偏“帮助模型判断”。

比如：

- 用户最近是否睡觉
- 当前时间
- 用户最近对话摘要
- 用户设定的主动推送规则
- Fitbit sleep context

它不一定直接成为推送内容，但会影响 LLM 判断。

面试可以这样说：

> 我把外部数据源拆成 alert/content/context 三类通道。alert 解决高优先级直通，content 解决候选内容流，context 提供低频背景信息。这样数据采集层和 Agent 决策层是解耦的，新增数据源只需要符合 MCP 输出协议，不需要改主动决策核心。

**8. [[多层门控]]：为什么不会乱发**
主动链路最怕的问题是：Agent 变成骚扰机器。

所以它有多层 gate。

可以概括成：

```text
冷却 gate
busy gate
presence gate
AnyActionGate
dedupe gate
quota gate
LLM decision gate
```

具体意思：

**冷却 gate**
刚推送过，就不要立刻再推。

**busy gate**
如果被动链路正在处理用户消息，主动链路不要插进来。

**presence gate**
看用户最近活跃状态，判断是否适合打扰。

**AnyActionGate**
控制后台自主动作的概率、每日次数、最小间隔。

**dedupe gate**
避免同一类内容重复推送。

**quota gate**
限制一天主动行为次数，防止过载。

**LLM decision gate**
最后让模型判断“这条内容是否真的值得主动说”。

面试里可以强调：

> 主动推送不是模型想发就发，而是先经过规则门控，再经过 LLM 价值判断，最后经过去重和配额控制。

**9. AgentTick：一次主动判断的执行单元**
`ProactiveLoop` 是循环，`AgentTick` 是每次循环真正做事的单元。

一次 tick 可以理解成：

```text
收集状态
 -> 拉取数据
 -> 构造 prompt
 -> 暴露 proactive 专用工具
 -> LLM 选择动作
 -> 执行动作
```

LLM 不只是输出自然语言，它通常会通过工具调用表达决策，比如：

```text
reply     -> 主动发消息
skip      -> 跳过
ack       -> 标记某条内容已处理
get_content -> 读取某条内容正文
```

这让主动链路可控，因为模型不是随便写一段话，而是在预定义动作空间里做选择。

**10. TurnOrchestrator：统一发送出口**
如果主动链路决定发消息，它不会自己直接调 Telegram/QQ。

而是交给 `TurnOrchestrator`。

它负责：

- 把 proactive 消息写入 session
- 记录这条消息是主动推送
- 附带 evidence/source_refs
- 通过统一 outbound port 发送
- 成功后更新 presence
- 失败后执行 failure side effect

这点很重要，因为主动链路和被动链路最终都归到统一的消息出口，方便审计、追踪和持久化。

**11. Drift：没有内容时也能工作**
如果本轮 tick 没有值得推送的内容，系统可以进入 Drift。

Drift 不是发消息，而是做后台任务，比如：

- 整理上下文
- 生成日报草稿
- 审计记忆
- 补全用户画像
- 根据 `SKILL.md` 执行自定义任务

这让主动链路形成闭环：

```text
有重要内容 -> 主动推送
没有重要内容 -> 后台探索/整理
用户发消息 -> 回到被动链路
```

所以项目描述里说“三链路闭环”：

```text
被动回复
主动推送
自主探索
```

主动链路连接了后两个：先尝试主动推送，没内容就进入 Drift。

**12. 面试讲法**
你可以这样完整讲：

> 主动链路由 ProactiveLoop 驱动，不是固定 cron，而是根据用户活跃度、电量模型、内容新鲜度和近期上下文动态计算 tick 间隔。每次 tick 先通过 Sensor 感知用户状态，再通过 DataGateway 并行拉取 alert/content/context 三路 MCP 数据。之后经过冷却、busy、presence、quota、AnyActionGate、去重等多层门控，最后由 LLM 判断是否值得主动触达。如果需要推送，则通过 TurnOrchestrator 统一写入 session 并发送；如果没有合适内容，则进入 Drift 执行后台任务。这个设计让 Agent 既能主动，又不会过度打扰用户。

简短版可以放简历：

```text
主动链路：设计 ProactiveLoop + AgentTick 主动推送架构，基于电量衰减模型、内容新鲜度与近期上下文计算 base_score，自适应调整 tick 间隔；通过 alert/content/context 三路 MCP 数据预取、多层门控与 LLM 决策实现低打扰主动触达，并在无推送内容时切换 Drift 后台任务。
```
